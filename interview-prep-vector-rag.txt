========================================================================
  축구 뉴스 프로젝트 - 벡터 DB & RAG 면접 준비 가이드
========================================================================

[ 기술 스택 ]
- TypeScript / Node.js (Express)
- Claude API (Anthropic) — 기사 요약 + RAG 답변 생성
- OpenAI text-embedding-3-small — 임베딩 (512차원)
- sqlite-vec — 벡터 저장/KNN 검색 (SQLite 확장)
- better-sqlite3 — 관계형 데이터 저장
- cheerio — 웹 크롤링


========================================================================
  1단계: 프로젝트 소개 (30초)
========================================================================

"축구 뉴스를 자동 수집/요약하고, 시맨틱 검색과 RAG 기반 질의응답이
가능한 파이프라인을 만들었습니다. RSS로 영문 기사를 수집하면 Claude가
한국어로 요약하고, OpenAI 임베딩으로 벡터화해서 sqlite-vec에 저장합니다.
사용자가 '손흥민 최근 활약'이라고 질문하면, 벡터 유사도로 관련 기사를
찾고, 그 기사를 컨텍스트로 Claude에게 넘겨 답변을 생성합니다."


========================================================================
  2단계: 벡터 DB를 왜 썼나요?
========================================================================

답변:
"기존에는 SQL WHERE category = '이적/영입' 같은 정확매칭 검색만
가능했습니다. 하지만 사용자가 '맨유 새 공격수'라고 검색하면 '이적/영입'
카테고리에 있지만 키워드가 일치하지 않는 기사를 못 찾아요.
벡터 검색을 도입하면 의미적으로 유사한 기사를 찾을 수 있습니다."

꼬리 질문 — "LIKE 검색이나 Full-Text Search로도 되지 않나요?"

"FTS는 형태소 기반이라 '이적'과 '영입'이 같은 맥락이라는 걸 모릅니다.
또 영문 원문을 한국어 요약으로 변환한 상태라, 사용자가 영어든 한국어든
자연스럽게 검색하려면 의미 기반 검색이 필요했습니다."


========================================================================
  3단계: 기술 스택 선택 근거
========================================================================

------------------------------------------------------------------------
  왜 OpenAI 임베딩?
------------------------------------------------------------------------

"Anthropic은 자체 임베딩 API가 없어서 별도 모델이 필요했습니다.
OpenAI text-embedding-3-small을 선택한 이유는 세 가지입니다:

1. 차원 수 조절 가능 — 1536 기본인데 512로 줄여 저장 비용 66% 절감
2. 배치 API 지원 — 여러 기사를 한 번에 임베딩해서 API 호출 횟수 최소화
3. 비용 — 1M 토큰당 $0.02, 기사 수천 건이면 거의 무료"

------------------------------------------------------------------------
  왜 sqlite-vec?
------------------------------------------------------------------------

"세 가지 벡터 DB를 비교했습니다:

- ChromaDB — 별도 Python 서버 필요, 관리 포인트 증가 → 탈락
- LanceDB — 좋지만 데이터가 SQLite와 별도 저장 → 관리 분산
- sqlite-vec — 기존 SQLite DB에 확장만 추가, DB 파일 하나로 모든 데이터 통합

기사 수천 건 규모에서는 HNSW 같은 근사 인덱스가 불필요하고,
brute-force KNN으로 충분히 빨라서 sqlite-vec가 최적이었습니다."

------------------------------------------------------------------------
  왜 512차원?
------------------------------------------------------------------------

"원래 1536차원인데, OpenAI의 Matryoshka Representation Learning
덕분에 차원을 줄여도 품질 저하가 적습니다. 512차원으로 줄이면:

- 벡터당 6KB → 2KB (저장 66% 절감)
- KNN 검색 속도 약 3배 향상
- 뉴스 요약 검색 수준에서는 품질 차이 미미"


========================================================================
  4단계: RAG 구조 설명 (화이트보드용)
========================================================================

질문: "최근 이적 시장 큰 뉴스가 뭐야?"
          |
          v
  +-- 임베딩 모델 --+
  | OpenAI API      |    같은 모델로 변환해야
  | 512차원 벡터     |    벡터 공간이 일치함
  +--------+--------+
           |
           v
  +-- sqlite-vec ---+
  | KNN 검색        |    코사인 거리 기준
  | WHERE MATCH     |    상위 5개 반환
  | ORDER BY dist   |
  +--------+--------+
           |
           v
     관련 기사 5개
     (id, title, summary, category)
           |
           v
  +-- Claude API ---+
  | System: 축구 전문 AI  |
  | User:            |
  |  "참고 기사:      |
  |   [기사1] ...     |  <-- 검색된 기사가 컨텍스트
  |   [기사2] ...     |
  |   질문: ..."      |
  +--------+---------+
           |
           v
     최종 답변 생성
  + 출처 기사 목록 반환

핵심 설계 원칙:
"RAG 프롬프트에 '제공된 기사에 없는 내용은 추측하지 마세요'라는
제약을 걸었습니다. 이게 hallucination 방지의 핵심입니다.
그리고 응답에 출처 기사 목록을 같이 반환해서
사용자가 검증할 수 있게 했습니다."


========================================================================
  5단계: 전체 파이프라인 흐름
========================================================================

[ 저장 시 ]

RSS 피드 → 크롤링(cheerio) → Claude 요약/분류
  → OpenAI 임베딩 생성 → sqlite-vec 벡터 저장 + SQLite 메타데이터 저장

[ 검색 시 (RAG) ]

사용자 질문 → OpenAI 임베딩 → sqlite-vec KNN 검색
  → 관련 기사 5개 조회 → Claude에게 기사+질문 전달 → 최종 답변 생성


========================================================================
  6단계: 예상 꼬리 질문 + 답변
========================================================================

------------------------------------------------------------------------
  Q. 임베딩 품질이 떨어지면 어떡하나요?
------------------------------------------------------------------------

"두 가지 계층으로 대응합니다.
첫째, 원문이 아니라 Claude가 요약한 한국어 텍스트를 임베딩합니다.
요약이 노이즈를 걸러주는 전처리 역할을 하는 셈이죠.
둘째, 임베딩 대상을 title + summary로 결합해서
제목의 키워드와 요약의 의미를 동시에 잡습니다."

------------------------------------------------------------------------
  Q. 기사가 10만 건이면?
------------------------------------------------------------------------

"현재 sqlite-vec는 brute-force KNN이라 O(n) 검색입니다.
10만 건이면 수십ms 정도로 아직 괜찮지만, 100만 건 이상이면
LanceDB나 Pinecone 같은 HNSW 기반으로 마이그레이션하거나,
카테고리별로 벡터 테이블을 분리해서 검색 범위를 줄이는
pre-filtering 전략을 쓸 겁니다."

------------------------------------------------------------------------
  Q. 기존 기사는 임베딩을 어떻게 했나요?
------------------------------------------------------------------------

"backfill 스크립트를 별도로 만들었습니다.
findArticlesNeedingEmbeddings()로 임베딩 미생성 기사를 조회하고,
OpenAI batch API로 50건씩 일괄 처리합니다.
rate limit 방어로 배치 사이 1초 딜레이도 넣었고요."

------------------------------------------------------------------------
  Q. 임베딩과 요약이 둘 다 실패하면?
------------------------------------------------------------------------

"요약 실패 시 status='failed'로 마킹되어 임베딩 단계에 도달하지 않습니다.
요약은 성공했는데 임베딩만 실패하면, 기사 자체는 정상 저장되고
임베딩만 빠집니다. backfill 스크립트가 LEFT JOIN으로 임베딩 없는 기사를
찾아 재처리하니까 데이터 손실은 없습니다."

------------------------------------------------------------------------
  Q. RAG 답변 품질을 어떻게 측정하나요?
------------------------------------------------------------------------

"현재는 수동 검증만 하고 있습니다. 개선한다면 Python RAGAS 프레임워크로
faithfulness(기사 근거 충실도), relevancy(질문 관련도),
context_recall(검색 정확도)을 자동 측정하는 평가 파이프라인을
추가할 계획입니다."

→ 모르는 건 솔직히 말하고 + 다음 단계를 제시하면 오히려 좋은 인상

------------------------------------------------------------------------
  Q. 왜 파이썬 안 썼나요?
------------------------------------------------------------------------

"프로젝트 전체가 TypeScript 단일 스택이고, LLM/임베딩/벡터검색 모두
TS SDK로 충분했습니다. 파이썬을 추가하면 런타임이 2개가 되어 배포와
유지보수 복잡도만 올라가는데, 얻는 이득이 없었습니다.
만약 커스텀 모델 학습이나 RAG 품질 평가(RAGAS)가 필요했다면
파이썬을 별도 서비스로 분리했을 겁니다."

========================================================================
  핵심 개념 정리 (빠르게 복습용)
========================================================================

[ 임베딩이란? ]
텍스트를 숫자 배열(벡터)로 변환하는 것.
의미가 비슷한 문장 → 비슷한 벡터 → 거리가 가까움.
예: "맨유 새 공격수 영입" → [0.02, -0.45, 0.13, ..., 0.89]  (512개 숫자)

[ 벡터 DB란? ]
이 벡터들을 저장하고, 주어진 벡터와 가장 가까운 것들을
빠르게 찾아주는 저장소. (KNN = K-Nearest Neighbors)

[ RAG란? ]
Retrieval-Augmented Generation.
벡터 DB에서 관련 정보를 검색(Retrieval)해서
LLM에게 컨텍스트로 제공(Augmented)하고
답변을 생성(Generation)하는 패턴.

LLM이 학습하지 않은 최신 정보(우리 뉴스 DB)를
답변에 활용할 수 있게 해주는 핵심 기술.

[ 벡터 DB vs 관계형 DB ]
- 관계형 DB: WHERE name = '손흥민'  → 정확히 일치하는 것만
- 벡터 DB:   MATCH '손흥민 활약'    → 의미적으로 비슷한 것 전부

[ Hallucination 방지 ]
RAG 프롬프트에 "제공된 기사만 참고하세요"라는 제약을 걸어
LLM이 없는 내용을 지어내는 것을 방지.
+ 출처(sources)를 같이 반환해서 사용자가 검증 가능.
